# 分销商城系统需求分析报告

**版本号：** V1.0  
**日期：** 2026 年 03 月  
**状态：** 初稿

---

## 1. 项目引言

### 1.1 项目背景
旨在构建一个支持商家自主运营、消费者购物、用户参与分销推广的综合性电商平台。通过返利机制激励用户分享商品，实现裂变式增长。

### 1.2 建设目标
1.  **商家端**：实现商品管理、订单处理、分销规则配置及财务结算。
2.  **用户端**：提供流畅的购物体验，支持一键生成推广素材，查看返利收益。
3.  **系统端**：确保订单状态流转准确，返利计算自动化、透明化，保障资金安全。

---

## 2. 用户角色定义

系统主要包含三种角色身份，其中“消费者”与“分销商”通常基于同一账号体系，根据权限或状态区分。

| 角色 | 别名 | 核心职责 | 权限说明 |
| :--- | :--- | :--- | :--- |
| **商家 (Admin)** | 平台管理员 | 运营、管理、结算 | 商品上下架、订单发货、返利比例设置、提现审核、用户管理。 |
| **消费者** | 普通买家 | 购物、售后 | 浏览商品、下单支付、确认收货、申请售后、查看个人订单。 |
| **分销商** | 推广员 | 推广、获利 | 生成邀请码/海报、分享商品、查看佣金明细、申请提现。<br>*(注：通常消费者满足一定条件可自动或申请成为分销商)* |

## 3. 功能需求详解

### 3.1 用户中心模块
*   **注册/登录**：支持手机号、微信授权登录。
*   **个人信息**：修改昵称、头像。
*   **收货地址**：增删改查，设置默认地址。
*   **分销资格**：
    *   查看分销状态（是否已开通）。
    *   申请成为分销商（如需审核）。
*   **钱包/资产**：
    *   查看累计收益、可提现金额、冻结金额。
    *   提现申请（绑定微信/支付宝/银行卡）。
    *   收支明细记录。

### 3.2 商品中心模块 (商家端)
*   **商品管理**：
    *   发布商品（标题、详情、规格、库存、价格）。
    *   商品上下架。
    *   **分销配置**：设置该商品的分销佣金比例（支持按百分比或固定金额）。
*   **分类管理**：商品类别的增删改查。

### 3.3 订单中心模块
*   **购物流程**：
    *   加入购物车、直接购买。
    *   **绑定关系**：下单时自动关联分销商（通过邀请码或链接进入）。
    *   提交订单、在线支付。
*   **订单状态机**：
    *   `待支付` -> `待发货` (支付后) -> `待收货` (发货后) -> `已完成` (确认收货/自动确认) -> `已结算` (过售后期)。
    *   异常状态：`已取消 `、` 退款中 `、` 已退款`。
*   **商家操作**：订单查询、发货（录入物流单号）、处理退款。
*   **消费者操作**：确认收货、申请售后、查看物流。

### 3.4 分销与返利模块 (核心)
*   **推广素材**：
    *   生成个人专属邀请码/邀请链接。
    *   生成个人专属二维码。
    *   生成商品专属海报（含商品二维码 + 分销商信息）。
*   **关系绑定**：
    *   支持“永久绑定”或“保护期绑定”（如 15 天内有效）。
    *   绑定逻辑：点击链接即绑定 vs 下单支付后绑定（建议后者，防刷）。
*   **返利计算**：
    *   触发节点：用户**确认收货**后。
    *   计算公式：`商品实付金额 * 分销比例 = 佣金`。
    *   状态管理：计算后进入`冻结期`（如 7 天无理由退货期内），期满无售后转为`可提现`。
*   **售后逆向流程**：
    *   若订单发生退款/退货，系统自动扣回已发放或冻结中的对应佣金。

### 3.5 后台管理模块 (商家)
*   **数据看板**：今日销售额、新增用户、新增分销商、待结算佣金。
*   **订单管理**：全量订单查询、导出。
*   **财务审核**：审核分销商的提现申请，打款操作。
*   **系统设置**：
    *   全局分销开关。
    *   自动确认收货天数设置。
    *   佣金冻结天数设置。

## 4. 核心业务流程图 (逻辑描述)

### 4.1 分销关系绑定流程
1.  用户 A (分销商) 生成商品链接/二维码。
2.  用户 B (消费者) 点击链接/扫码。
3.  系统记录 B 的访问来源为 A (写入临时缓存或数据库)。
4.  用户 B 下单并**支付成功**。
5.  系统正式将 B 绑定为 A 的下级 (记录 `order_relation` 表)。

### 4.2 返利结算流程
1.  用户 B 点击“确认收货”或系统超时自动确认。
2.  订单状态变更为 `已完成`。
3.  系统触发返利计算任务：
    *   查找 B 的分销商 A。
    *   读取商品设定的返利比例。
    *   计算佣金金额。
    *   创建 `rebate_record`，状态为 `冻结中`。
4.  进入售后保护期（例如 7 天）。
5.  保护期结束且无退款申请：
    *   `rebate_record` 状态变更为 `可提现`。
    *   更新 A 的钱包余额。
6.  若保护期内发生退款：
    *   `rebate_record` 状态变更为 `已失效`。
    *   若已入账，则从 A 钱包扣减（若余额不足则记为负数，待后续抵扣）。

## 5. 数据需求 (核心实体)

1.  **Users (用户表)**: ID, 手机号，密码，角色标识，分销商 ID，分销状态，钱包余额。
2.  **Products (商品表)**: ID, 名称，价格，库存，**分销佣金比例**, 状态。
3.  **Orders (订单表)**: ID, 用户 ID, 总金额，支付状态，物流状态，**分销商 ID**, 创建时间。
4.  **Order_Items (订单明细表)**: ID, 订单 ID, 商品 ID, 数量，单价。
5.  **Rebate_Records (返利记录表)**: ID, 分销商 ID, 关联订单 ID, 佣金金额，状态 (冻结/可用/已提现/已失效), 创建时间，结算时间。
6.  **Withdraw_Requests (提现申请表)**: ID, 用户 ID, 金额，收款账号，状态 (审核中/通过/拒绝), 审核时间。

## 6. 非功能需求

1.  **安全性**：
    *   支付接口需加密，防止篡改金额。
    *   用户敏感信息（手机号、身份证）脱敏存储。
    *   防刷机制：同一设备/IP 频繁注册或下单需触发风控。
2.  **性能**：
    *   支持高并发访问（特别是秒杀或推广高峰期）。
    *   返利计算建议采用**异步队列**处理，避免阻塞订单确认流程。
3.  **一致性**：
    *   订单退款与佣金扣回必须保证数据强一致性（使用事务）。
4.  **易用性**：
    *   分销海报需支持自定义模板，生成速度快。
    *   移动端适配（小程序/H5/App）。
